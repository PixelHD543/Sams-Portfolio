  • ✅ Physical demonstration on 1/8th scale nitro RC car

System Architecture – Distributed Control

The system uses a distributed architecture where each microcontroller hosts its own HTTP server, and Unreal Engine cycles through vehicle webpages to read/write data.

┌─────────────────┐      UDP/HTTP      ┌─────────────────┐
│  Unreal Engine  │ ◄────────────────► │  ESP32 S3       │
│  (Simulation)   │      (WebSocket)   │  (Vehicle 1)    │
└─────────────────┘                    └─────────────────┘
         │                                       │
         │ HTTP (cycle through IPs)              │
         ▼                                       ▼
┌─────────────────┐                    ┌─────────────────┐
│  ESP32 S3       │                    │  RPLidar        │
│  (Vehicle 2)    │                    │  Servo Control  │
└─────────────────┘                    └─────────────────┘

Why each vehicle hosts its own server: If Unreal Engine operated as a central server, it would maintain high-bandwidth data streams to every connected vehicle, causing performance degradation. Instead, each ESP32 hosts its own HTTP page – Unreal Engine cycles through server webpages, reading values and writing commands, then continues to the next server.

LiDAR Navigation – 16K Samples/Second

The RPLidar provides distance and rotation measurements at 16,000 samples per second. Unreal Engine processes these to create real-time navigation boundaries.

// LiDAR data processing in Unreal Engine (conceptual)
// Spawn oblong actors equal to LiDAR rotation + distance measurement
// 16,000 samples/sec → 16,000 boundary actors per second
// Plugin creates navigable area within accumulated boundaries

function processLiDARScan(rotation, distance) {
  spawnBoundaryActor(rotation, distance);
  accumulatedScans++;
  if (accumulatedScans >= 16000) {
    updateNavigationMesh();  // Recalculate drivable area
    accumulatedScans = 0;
  }
}

Performance optimisation: 16K samples/second creates precise area mapping but reduces performance. The system allows users to adjust sample rate dynamically – balancing resolution against hardware capabilities for universal accessibility.

ESP32 Control – Modular Vehicle Support

The ESP32 S3 (240MHz) receives LiDAR data, processes navigation commands, and controls vehicle servos. The system supports multiple steering methods:

Technical Challenges & Solutions

Challenge 1: ESP32 Upload Issues
The ESP32's open-source nature means no official compiler. Arduino IDE required, but library changes during development separated UART and USB port functions. The UART port receives data from IDE, USB port handles Serial.print().

Solution: After correcting tool settings, the program uploads successfully. The ESP32 performs as intended.

Challenge 2: Program Won't Run Without IDE Connected
The ESP32 pauses at Serial.print() sections unless connected to Arduino IDE.

Solution: Remove debug information for production – the device works when debug prints are removed or when powered independently.

Challenge 3: Performance vs. Resolution Trade-off
16K LiDAR samples/second creates precise mapping but reduces performance.

Solution: Sample rate stored as variable – users adjust resolution based on their hardware capabilities, making software universally accessible.

Challenge 4: Unreal Engine Error Messages
Block-based programming (Blueprint) produces vague error messages without resolution descriptions.

Solution: UI code is relatively simple; web server plugin handles HTTP connections independently.
