What This Project Taught Me About Git & GitHub
  • 🔀 Branching strategy: Used main for production, dev for testing new features, and gh-pages for deployment. Simulated hardware revision control.
  • 🤖 GitHub Actions CI/CD: Automated linting + deployment on every push. No manual FTP — pure pipeline automation.
  • 📈 Project tracking & Issues: Used GitHub Projects to track client feedback (grandma's change requests!) — analogous to tracking bugs in VHDL simulation or PLC logic errors.
  • 📦 Static site generation & versioning: Understood how to rollback to any previous commit, similar to rolling back an FPGA bitstream or firmware release.
  • 🔐 Access control & collaboration: Gave my grandma read access to the repo to view progress — mirrors team-based embedded development where designers, testers, and managers interact via PRs.
🔌 Direct Link to Electrical & Embedded Engineering:

Git isn't just for websites. In FPGA/ASIC design, we use Git to version control .vhd/.sv files, track synthesis/P&R reports, and collaborate on IP cores. For embedded systems (STM32, ESP32, ARM), Git tracks firmware revisions, RTOS configs, and binary releases. The same branching and CI/CD concepts I used here translate to automated hardware tests, LoRaWAN node firmware updates, and PLC code versioning in TIA Portal or Codesys.

How This Applies to My Target Roles (FPGA, Embedded, LoRa, PLCs)

FPGA Development

Version control for Verilog/VHDL modules, testbenches, and constraint files. Git hooks for pre-commit syntax checking. GitHub Actions to run simulation regressions (Icarus Verilog / GHDL). Managing multiple branches for different clock domains or IP variants.

Embedded Systems (C/RTOS)

Firmware versioning, semantic tagging for releases, and automated builds via CMake + GitHub Actions. Issue tracking for hardware-in-the-loop bugs. Using Git submodules for middleware or vendor SDKs (STM32Cube, Zephyr).

RF / LoRa Communication

Managing configuration profiles for different frequency bands (EU868, US915) using Git branches. Keeping node firmware and gateway code in sync. Documenting over-the-air update (OTAA) credentials securely via GitHub Secrets. Collaborative debugging of LoRaWAN packet formats using PR comments.

PLC & Industrial Automation

Versioning TIA Portal projects, Codesys code, and WinCC HMI panels. Using Git LFS for large binary PLC archives. Change logs for safety-rated logic modifications (IEC 61131-3). Simulating CI pipelines with PLCSim Advanced to test on every commit.

My Standard Git Workflow (Used Here & for Electronics)
1. git checkout -b feature/lora-packet-parser   # new branch for specific task
2. (hack, commit, push)
3. Open Pull Request → run automated checks (syntax / simulation)
4. Code review with team (or myself, simulated)
5. Merge to dev → integration tests
6. Tag release: git tag -a v1.2.0 -m "Add OTAA support"
7. Deploy/Generate firmware binary or bitstream
8. Rollback if needed: git revert or checkout previous tag

This exact workflow was used for the website. For my NATO flood early warning system, I'm using the same pattern: feature branches for each LoRaWAN sensor driver, CI that checks compilation for ARM, and tagged releases for field-deployed nodes.
