Project Overview – Phones-R-Us

Phones-R-Us, a mobile phone manufacturer, needed to automate a manual pick-and-place station to handle mPhone 26 units in three colour variants: Crimson Red, Obsidian Black, and Quantum Blue. The target throughput was one phone every two seconds.

Key requirements:

  • ✅ 5+ DOF robot for future flexibility
  • ✅ Colour-based sorting (red/black/blue boxes)
  • ✅ Defective unit rejection bin
  • ✅ Full box moving capability (63 phones ≈ 13kg)
  • ✅ Collaborative operation preferred (no enclosure)
  • ✅ 3m × 3m workspace constraint

CoppeliaSim Simulation – State Machine & Lua Control

A complete functional simulation was developed in CoppeliaSim EDU 4.10, implementing an 8-state finite state machine for the robot controller.

State Machine Design

State 0: Idle – waiting for phone
State 1: Move to approach point (pick)
State 2: Move to pick position
State 3: Attach phone (vacuum on)
State 4: Move to retreat point (pick)
State 5: Move to approach point (place)
State 6: Move to place position (colour-dependent)
State 7: Release phone (vacuum off)
State 8: Move to retreat point (place) → return to State 0

Waypoint-Based Motion (Collision-Free)

-- Waypoint sequence for pick operation
waypoints = {
  {x = 0.800, y = 1.200, z = 0.850},  -- Approach
  {x = 0.800, y = 1.200, z = 0.700},  -- Pick
  {x = 0.800, y = 1.200, z = 0.900},  -- Retreat
}

function moveThroughWaypoints(waypoints)
  for i, wp in ipairs(waypoints) do
    setGoalPosition(wp)
    while not stepToward() do
      simIK.handleGroup(ikEnv, ikGroup)
    end
  end
end

Colour Detection & Routing

function getPhoneColour(r, g, b)
  -- Crimson Red: high red, low green and blue
  if r > 0.7 and g < 0.3 and b < 0.3 then return "red" end
  -- Obsidian Black: very dark
  if r < 0.2 and g < 0.2 and b < 0.2 then return "black" end
  -- Quantum Blue: high blue
  if b > 0.5 and r < 0.4 and g < 0.4 then return "blue" end
  return "reject"
end

-- Route to correct box
if colour == "red" then
  placePos = redBoxPos
elseif colour == "black" then
  placePos = blackBoxPos
elseif colour == "blue" then
  placePos = blueBoxPos
else
  placePos = rejectBinPos
end

Dual-Mode Gripper – Phones (200g) + Full Boxes (13kg)

The intermediate specification required "proven capability to pick and place phones AND move full boxes". A dual-mode gripper was designed:

200g payload

Box Mode: Parallel jaws
13kg payload
Spring-engaged fail-safe

Box Moving Sequence (Lua)

function moveFullBox(boxColour)
  setGripperMode("box")  -- Switch to parallel jaws
  moveToPosition({boxPos.x, boxPos.y, boxPos.z + 0.1})  -- Approach
  moveToPosition(boxPos)  -- Pick position
  closeGripper()
  waitForGripConfirmation()
  moveToPosition({boxPos.x, boxPos.y, boxPos.z + 0.2})  -- Lift
  moveToPosition(palletPos)  -- Move to collection area
  openGripper()
  setGripperMode("phone")  -- Switch back
end

Safety Assessment – ISO 10218-1 Compliant

A systematic hazard analysis identified 7 primary hazards. Each was mitigated following the hierarchy of controls.

Key Safety Features Implemented in Simulation

  • Light curtain (simulated): Perimeter safety zone – robot stops within 20ms
  • Speed-limited zones: 250 mm/s within 500mm of operator
  • Emergency stops: Three locations (operator station, robot controller, conveyor)
  • Fail-safe gripper: Spring-engaged jaws (air-released) for box mode
  • Vacuum confirmation sensor: Verifies grip before motion
-- Speed-limited zone monitoring (Lua)
function checkSafetyZones(tipPos, operatorPos)
  local dist = distance(tipPos, operatorPos)
  if dist < 0.5 then
    sim.setIntegerSignal("emergency_stop", 1)
    sim.addStatusbarMessage("EMERGENCY STOP: Operator too close")
  elseif dist < 1.0 then
    sim.setFloatSignal("robot_speed_limit", 0.25)  -- 250 mm/s
    sim.addStatusbarMessage("WARNING: Speed reduced")
  else
    sim.setFloatSignal("robot_speed_limit", 1.0)
  end
end

Standards Compliance: ISO 10218-1:2011, ISO 13849-1:2015 (PLr = d), PUWER 1998, UKCA marking requirements.

Performance Evaluation – Cycle Time Analysis

Cycle Time Breakdown (per phone)

  • Pick: Approach → pick → retreat: 0.9s
  • Place: Approach → place → retreat: 0.9s
  • Conveyor wait + colour detection: 0.6s
  • Variation + safety zones: 0.4s
  • Realistic total: 2.8 seconds per phone

Target was 2.0 seconds – gap of 0.8 seconds (28% shortfall).

Critical Reflection – Why the Gap?

  • Sequential operation: Full pick-place-return cycle required for each phone
  • Waypoint-based motion: Approach/retreat waypoints add ~0.8s per cycle
  • Conveyor spacing: Simulated 5s auto-spawn is placeholder for real production
